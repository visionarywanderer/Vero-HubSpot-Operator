# 13 — Safety & Governance Rules

## Purpose
Non-negotiable rules enforced across every module. These protect client data, prevent accidental damage, and ensure full auditability.

## Priority: P0 | Dependencies: None (all modules must implement these)

---

## Rule 1: Sandbox-First for New Clients

**Every new client engagement starts in a sandbox portal.**

- First run of any operation against a new portal must use a HubSpot sandbox (available on Professional+)
- The orchestrator checks `portalConfig.environment` before any write operation
- If environment = `production` and it's the first session → block writes, warn user
- Only after sandbox validation can the portal be switched to production

```typescript
function canWrite(portalConfig: PortalConfig, isFirstSession: boolean): boolean {
  if (portalConfig.environment === 'sandbox') return true;
  if (portalConfig.environment === 'production' && isFirstSession) return false;
  return true;
}
```

---

## Rule 2: Dry-Run for All Bulk Scripts

**Every script generated by the Script Engine runs in dry-run mode first.**

- The Script Runner enforces `--dry-run` on first execution
- Dry-run output is shown to the user with exact counts and sample records
- Only after user reviews dry-run output can they choose `--execute`
- All scripts MUST support `--dry-run` flag (enforced in template)

```
Flow: Generate script → Human reviews code → Dry-run → Human reviews output → Execute → Log results
```

---

## Rule 3: Workflows Deploy Disabled

**All AI-generated workflows deploy with `isEnabled: false`.**

- The Workflow Engine validation rejects any spec with `isEnabled: true`
- After deployment, the user must enable the workflow manually in HubSpot UI
- This ensures a human verifies enrollment criteria and actions before any contacts are enrolled

```typescript
function validateWorkflowSafety(spec: any): string[] {
  const errors = [];
  if (spec.isEnabled !== false) {
    errors.push('SAFETY VIOLATION: isEnabled must be false. Workflows must be enabled manually.');
  }
  return errors;
}
```

---

## Rule 4: No Deletions Without Double Confirmation

**Record and workflow deletions require explicit confirmation with the exact ID.**

- Delete operations are flagged as `risk: 'high'` by the orchestrator
- User must type the exact record ID or workflow flow ID to confirm
- HubSpot does NOT support undo for workflow deletions
- Record deletions go to the recycle bin (30-day recovery) but AI should never rely on this

```
Orchestrator: "This will permanently delete workflow 'Lead Routing' (flowId: 1734596242).
              HubSpot cannot restore deleted workflows via API.
              Type the flowId to confirm: "
User: "1734596242"
→ Proceed with deletion
```

---

## Rule 5: Full Change Log

**Every change the app makes is logged.**

- The Change Logger (spec 05) is called by every module after every write operation
- Logs include: timestamp, portal, layer, module, action, recordId, before, after, status
- Logs are never deleted
- Logs support export to CSV for client reporting
- Before/after values enable manual rollback if needed

---

## Rule 6: Rate Limiting Is Mandatory

**All API calls respect HubSpot limits. No exceptions.**

| Endpoint Type | Limit | Enforcement |
|--------------|-------|-------------|
| Standard API | 100 requests / 10 seconds | Token bucket in API Client |
| Search API | 4 requests / second | Separate limiter |
| Batch API | 10 records / request | Enforced in batch helper |
| Script Layer | 120ms delay between requests | Built into script template |

- On 429 response: pause ALL requests for `Retry-After` seconds
- Never retry immediately on rate limit
- API Client enforces this globally — individual modules cannot bypass

---

## Rule 7: Human-in-the-Loop for Generated Code

**LLM generates. Human reviews. App executes. Never auto-execute.**

```
┌──────────┐     ┌───────────┐     ┌──────────┐     ┌──────────┐
│ LLM      │ ──→ │ Human     │ ──→ │ App      │ ──→ │ HubSpot  │
│ generates│     │ reviews   │     │ executes │     │ API      │
└──────────┘     └───────────┘     └──────────┘     └──────────┘
```

This applies to:
- Workflow v4 JSON specs → human reviews before POST
- Generated scripts → human reviews code before execution
- Property creation specs → human confirms before POST
- List filter criteria → human verifies before POST
- Any write operation with > 10 affected records

**Exception**: Read-only operations (audits, queries) can execute without confirmation.

## Rule 8: Scope Checking Before Operations

**The orchestrator checks available scopes before routing any operation.**

- If a required scope is missing, tell the user which scope they need to add
- Never attempt an API call that will fail due to missing scopes
- Store discovered scopes per portal (from token validation)

```typescript
const SCOPE_MAP = {
  'A1': ['crm.objects.contacts.read'],  // Read contacts
  'A2': ['crm.objects.contacts.write'], // Create contacts
  'B1': ['automation'],                  // Generate workflow
  'B2': ['automation'],                  // Deploy workflow
  'C2': ['crm.schemas.contacts.write'], // Create property
  'D1': ['crm.lists.write'],            // Create list
};

function checkScopes(module: string, availableScopes: string[]): boolean {
  const required = SCOPE_MAP[module] || [];
  return required.every(s => availableScopes.includes(s));
}
```

---

## Rule 9: No Sensitive Data in Prompts

- Never include actual API tokens in LLM prompts
- Never include personal email addresses or phone numbers in LLM prompts for script generation
- Use placeholder variables: `{token}`, `{ownerEmail}`, `{contactId}`
- Actual values are injected at execution time from portal config

---

## Rule 10: Version Control for Generated Artifacts

- Every generated workflow spec is saved with a timestamp
- Every generated script is saved with a timestamp
- This enables rollback and auditing of what was generated vs. what was deployed
- Storage structure:

```
artifacts/
  {portalId}/
    workflows/
      2026-03-07T14-30-00_lead-routing.json
      2026-03-07T15-00-00_stalled-deal-alert.json
    scripts/
      2026-03-07T16-00-00_name-standardization.js
      2026-03-07T16-00-00_name-standardization_dryrun.log
      2026-03-07T16-30-00_name-standardization_execute.log
```
